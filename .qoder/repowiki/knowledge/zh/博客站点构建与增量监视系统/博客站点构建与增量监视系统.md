本仓库采用基于 Python 的轻量级静态内容构建系统，将 posts/ 目录下的 Markdown 源文件编译为浏览器可直接加载的 JSON/JS 数据脚本，再由前端页面在运行时动态渲染。整个流程由两个核心脚本驱动：一次性构建器与轮询式增量监视器。

## 构建系统与工具链
- 构建语言：纯 Python（无第三方依赖），通过标准库 re、json、pathlib、shutil、subprocess 实现。
- 构建入口：tools/build_posts.py，扫描 posts/ 下所有分类目录中的 .md 文件，解析 Front Matter 与正文，生成 data/ 与 data/articles/<category>/<slug>.{js,json}。
- 增量监视：tools/watch_posts.py 以 1 秒为间隔轮询 posts/**/*.md 的文件时间戳与大小变化，触发 build_posts.py 重建；Windows 用户可通过 tools/start_post_watch.bat 双击启动。
- 文档说明：tools/README.md 提供构建命令、图片存放约定与碎片（Fragments）格式规范。

## 输入输出约定
输入：
- 文章 Markdown：posts/<category>/<slug>.md，支持 YAML 风格 Front Matter（自定义解析器，非 PyYAML）
- 碎片笔记：posts/fragments/fragments.md，使用 ## YYYY-MM-DD HH:MM:SS 二级标题分段
- 文章配图：image/<category>/<slug>/...，相对路径自动映射到该目录
- 碎片配图：image/Fragment/<year>/...，按年份子目录组织

构建产物：
- data/posts.json / data/posts.js — 文章索引（不含正文）
- data/categories.json / data/categories.js — 分类聚合
- data/fragments.json / data/fragments.js — 碎片列表
- data/articles/<category>/<slug>.json / .js — 单篇文章完整数据（含正文）

## 架构与关键设计决策
1. Front Matter 自定义解析器：不依赖外部 YAML 库，自行实现标量、数组、布尔、数字类型识别，并支持 - item 多行列表语法。
2. 双格式输出（JSON + JS）：同一份数据同时写出 .json 和 window.__BLOG_*__ = ... 形式的 JS 全局变量，兼顾 API 消费与浏览器内联加载两种场景。
3. 内容分离策略：索引文件剔除 content 字段以降低体积，正文仅保留在单篇 articles/<category>/<slug>.json 中，按需请求。
4. 片段日期规范化：将 YYYY-MM-DD HH:MM:SS 统一转换为 ISO 8601 带时区字符串，并提供 timeLabel 与 year 辅助字段。
5. 增量监视采用轮询而非文件系统事件：避免跨平台 watchdog/inotify 等依赖，用 stat().st_mtime_ns + st_size 双重校验保证可靠性。
6. 图片路径自动映射：Markdown 正文中的相对图片路径在构建期被替换为 image/<category>/<slug>/... 的实际地址，作者无需手动维护绝对路径。

## 开发者应遵循的规则
- 新增文章时在 posts/<category>/ 下创建 <slug>.md，并在 Front Matter 中至少提供 title、date；可选 tags、cover、featured、pinned、showInRecent、showInArchive 等字段控制展示行为。
- 文章配图必须放在 image/<category>/<slug>/ 目录下，正文中以相对路径引用。
- 碎片统一写入 posts/fragments/fragments.md，每个条目以 ## YYYY-MM-DD HH:MM:SS 二级标题开头，配图放 image/Fragment/<year>/。
- 修改内容后运行 py tools/build_posts.py 重新生成数据，或启动 py tools/watch_posts.py 进入热重载模式。
- 不要直接编辑 data/ 下的生成文件，它们会在下次构建时被覆盖。
